Проект внедрения Service Desk 2026

Новый Service Desk:
прозрачность, скорость, контроль

Структурированный учёт обращений, диспетчеризация, SLA по отделам, автоматизация маршрутизации и единый портал для всех подразделений.

Цель проекта

Внедрить централизованную структуру для всех обращений: учёт, диспетчеризация, анализ эффективности, выявление зон улучшения и оптимизация бизнес-процессов.

Критерии приёмки:
интеграция с почтой, кастомизация интерфейса и процессов, маршрутизация по категориям, поддержка иерархии.

Ключевые стейкхолдеры

  • Владелец процесса: Юлия Логонова (руководитель поддержки) – цели, KPI, SLA, процессы.
  • Агенты L1: обратная связь по usability, скорость обработки.
  • Бизнес-подразделения: 1С, Инженеры, Конструкторы – боли, типичные обращения.
  • IT/Администрирование: миграция данных, интеграции, API.

Ключевые метрики AS-IS

150–750 заявок/день 95% канал — почта

Текущая система: Redmine + кастомный плагин (без поддержки).
🔴 Узкие места: ручная маршрутизация, отсутствие SLA по группам, дублирование данных.

Функциональные требования (ключевые)
ФТ-1

Управление пользователями

6 ролей (инициатор, наблюдатель, исполнитель, руководитель, диспетчер, админ). SSO через ADFS, синхронизация из AD.

ФТ-2

Автоматическое создание заявок

Из писем на help@marya.ru (каждые 2 мин). Извлечение темы, тела, вложений, определение инициатора.

ФТ-3

Маршрутизация

Правила Тип→Группа, параллельное/последовательное назначение, возврат на 1 линию с комментарием.

ФТ-4

Статусы и ЖЦ

7 статусов (Новая, В распределении, В работе, В ожидании, Выполнена, Отклонена, Дубликат). Цветовая индикация.

ФТ-5

SLA и метрики

5 типов времени (реакция, диспетчеризация, реакция исполнителя, время в работе, время жизни). Остановка в нерабочее время.

ФТ-6

Оценки качества

ХОРОШО, ПЛОХО, ВЕРНУТЬ В РАБОТУ. 24ч на оценку, автозакрытие без оценки.

ФТ-7

Интеграции

Почта, 1С (структура, графики), ADFS, REST API, Webhook, Чат-бот.

ФТ-8

Кастомизация без кода

Настройка типов заявок, полей, приоритетов, правил маршрутизации, SLA через UI администратора.

Нефункциональные требования

Производительность

  • Время отклика < 2 сек (95% запросов)
  • Загрузка 1000 заявок < 3 сек
  • Пропускная способность: 750 заявок/день, 100 одновременных пользователей
  • Обработка email до 50 писем/мин

Безопасность

  • ADFS (SSO)
  • 6 ролей, принцип минимальных привилегий
  • HTTPS, маскирование ПД в логах
  • Аудит входов, хранение логов 1 год
  • Соответствие 152-ФЗ, хранение ПД в РФ

Доступность

  • 99.5% в рабочее время (8:00–22:00)
  • RTO < 4 часа, ежедневный бэкап БД
  • Мониторинг доступности каждую минуту

Масштабируемость

  • Горизонтальное масштабирование без простоя
  • Поддержка до 5000 активных пользователей
  • До 500 групп с иерархией
Бизнес-требования
Бизнес-правила SLA

Умный расчёт

  • SLA считается в рабочие часы подразделения
  • Остановка в нерабочее время, выходные, праздники
  • Независимые SLA для каждой группы в цепочке
  • Напоминания при 70% истечения, эскалация руководителю
Бизнес-процессы

TO-BE сценарии

  • Линейная задача: поступление → диспетчеризация → исполнение → оценка
  • Согласование: запрос согласующему, при отклонении — закрытие без оценки
  • Параллельное исполнение: связанные задачи для 2+ групп Группа1 → Группа2 → ... → ГруппаN
  • Анкета нового сотрудника: автоподстановка ПО/прав, создание нескольких заявок
Интеграции

С корпоративной экосистемой

  • Почта (IMAP/SMTP) – автосоздание заявок
  • 1С – структура подразделений, графики работы, пользователи
  • ADFS (SSO) – единый вход
  • REST API, Webhook, Чат-боты
Юридические требования

152-ФЗ и безопасность

  • Хранение персональных данных на территории РФ
  • Согласие на обработку ПД при первом входе
  • Право на удаление ПД по запросу
  • Хранение заявок 3 года, логов – 1 год

Какой сервис-деск выбрать? Опыт компаний
GLPI

Для кого: производство, госсектор, IT-аутсорс

  • Почему выбирают: объединяет help desk + учёт активов (CMDB) + лицензии + финансы в одной системе.
  • Идеально, если важно отслеживать не только заявки, но и каждый компьютер, станок, договор.
  • Сильная инвентаризация, активное сообщество, бесплатная версия.
iTop

Для кого: банки, страховые, крупный ритейл, ITIL-ориентированные

  • Почему выбирают: строгие ITIL-процессы, визуальный редактор процессов
  • Идеален для компаний, где нужен формальный подход к управлению инцидентами, проблемами, изменениями.
  • Поддержка сложных SLA и цепочек согласований.
Znuny / OTOBO

Для кого: крупные европейские корпорации, «наследники» OTRS (Open Ticket Request System)

  • Почему выбирают: стабильность, долгосрочная поддержка, знакомые ITIL-процессы.
  • Znuny — максимально близок к классическому OTRS, подходит для консервативных ИТ-отделов.
  • OTOBO — современный, с обновлённым интерфейсом и AI-функциями.
Zammad

Для кого: стартапы, e-commerce, IT-компании, клиентские сервисы

  • Почему выбирают: современный дизайн, многоканальность (email, чат, Telegram, соцсети) из коробки.
  • Лёгкость настройки, отличный UX, быстрое внедрение.
  • Подходит для команд, которые ценят простоту и скорость работы.
Более подходящие GLPI или iTop — оба решения поддерживают учёт активов, гибкую маршрутизацию, SLA по отделам, интеграцию с 1С и AD, а также кастомизацию без программирования.
План внедрения (MVP) и ключевые выгоды
  • Недели 1-4: выбор платформы, развёртывание
  • Недели 5-7: кастомизация типов, правил маршрутизации, SLA
  • Недели 8-9: интеграция с почтой, 1С, миграция данных
  • Недели 10-11: тестирование, обучение агентов и диспетчеров
  • Недели 12-16: пилотный запуск, сбор обратной связи, корректировки

Ключевые выгоды

  • 📈 Сокращение времени обработки заявок на 30%
  • 📊 Прозрачные SLA и дашборды по отделам
  • 🤖 Снижение ручного труда L1 на 40%
  • 🔒 Соответствие 152-ФЗ, хранение данных в РФ
  • 💰 Отказ от необслуживаемой системы Redmine
  • 🛠️ Кастомизация без привлечения разработчиков
